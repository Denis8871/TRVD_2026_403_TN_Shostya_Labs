**1. Обраний технологічний стек**

- **BaaS (Backend-as-a-Service): Supabase.**
  - *Чому:* Надає готову інфраструктуру: базу даних PostgreSQL, систему автентифікації та автозгенероване REST API, що дозволяє зосередитися на бізнес-логіці та UI/UX клієнта.
- **База даних: PostgreSQL (через Supabase).**
  - *Чому:* Потужна реляційна СКБД із підтримкою Row Level Security (RLS) для безпечного розмежування доступу користувачів до своїх даних на рівні самої БД.
- **Front-end: React 18 + React Router + Vite.**
  - *Чому:* Створення швидкого Single Page Application (SPA), зручна маршрутизація, ефективне оновлення інтерфейсу та миттєва збірка проєкту.
- **Клієнт API: @supabase/supabase-js.**
  - *Чому:* Офіційний SDK для взаємодії з базою даних та системою авторизації Supabase без необхідності ручного налаштування HTTP-запитів через Axios чи Fetch.
- **Безпека: Supabase Auth (JWT) + Row Level Security (RLS).**
  - *Чому:* Забезпечує надійну реєстрацію/логін та апаратно гарантує, що запити до БД повертатимуть лише ті задачі, які належать поточному авторизованому користувачу.

**3. Проєктування БД (ER-Model)**

Особливість Supabase полягає в тому, що він має власну захищену схему auth для зберігання облікових даних. Ми створюємо публічну таблицю profiles (у схемі public), яка зв'язується з auth.users.

Фрагмент кода

erDiagram

`   `AUTH\_USERS ||--|| PROFILES : "has profile"

`   `PROFILES ||--o{ TASKS : "creates"



`   `AUTH\_USERS {

`       `uuid id PK "Supabase Auth ID"

`       `string email UK

`       `string encrypted\_password

`       `datetime created\_at

`   `}



`   `PROFILES {

`       `uuid id PK "FK to auth.users.id"

`       `string role "user, admin"

`       `string full\_name

`       `boolean is\_blocked "Status for Admin Panel"

`       `datetime updated\_at

`   `}



`   `TASKS {

`       `uuid id PK

`       `uuid user\_id FK "References profiles.id"

`       `string title

`       `string description

`       `string status "todo, done"

`       `string priority "low, medium, high"

`       `datetime due\_date

`       `datetime created\_at

`   `}

**Ключі і зв'язки:**

- **auth.users -> public.profiles (1:1):** Користувач створюється в системі Supabase Auth, після чого автоматично (за допомогою написаного SQL Trigger handle\_new\_user()) створюється його публічний профіль у таблиці PROFILES.
- **PROFILES -> TASKS (1:N):** Один користувач може створити багато задач.
- id у всіх таблицях використовує формат UUID, що є стандартом та найкращою практикою для PostgreSQL у Supabase.

**Безпека даних (Row Level Security - RLS)**

Для таблиці TASKS активовано RLS з наступними політиками:

- **SELECT:** Користувач може бачити лише рядки, де user\_id = auth.uid().
- **INSERT:** Користувач може створювати задачі лише з власним user\_id.
- **UPDATE/DELETE:** Користувач може змінювати/видаляти лише власні задачі.
